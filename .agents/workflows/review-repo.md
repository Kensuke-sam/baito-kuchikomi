# レポジトリ全体のレビューと修正

このワークフローは、レポジトリ全体をスキャンしてコードの品質、セキュリティ、パフォーマンスなどの改善点を提案し、必要に応じて実際の修正を行うための手順です。

1. **現状の把握とプロジェクト構造の確認**
   - ルートディレクトリや主要なソースディレクトリ（`app/`, `src/`, `components/`, `lib/` など）の構成を `list_dir` や `find_by_name` を使って確認します。
   - `package.json`、`tsconfig.json`、その他の設定ファイルを読み込み、プロジェクトの技術スタックとコーディング規約を把握します。

2. **Lintスクリプトや型チェックの実行**
   - `package.json` に定義されているLintや型チェックコマンド（例: `npm run lint`, `npx tsc --noEmit` など）を `run_command` を使って実行し、機械的に検知できるエラーや警告を洗い出します。
   - `// turbo` アノテーションをつけて自動実行できるものは行います。

3. **コードの静的レビュー**
   - エラーが出ているファイルや、複雑なロジックを含む重要なファイルを `view_file` で読み込みます。
   - 以下の視点でコードの改善点を直視的に分析します：
     - パフォーマンス（不要な処理、非効率なループ、重いレンダリングなど）
     - セキュリティ面
     - 可読性と保守性（DRY原則、関数分割の必要性、適切な命名など）
     - Next.jsなどのフレームワークのベストプラクティスに沿っているか

4. **改善案の提示とユーザーへの確認（必須）**
   - 洗い出した改善点と、それらに対する具体的な修正アプローチをリストアップします。
   - **【重要】** ユーザーへの確認事項やシステム上のステータス説明は、**必ずすべて日本語で**分かりやすく提示してください。英語のメッセージがそのまま表示されないように配慮します。

5. **コードの修正とリファクタリング**
   - ユーザーから修正の承認を得た箇所について、ファイル書き換えツール（`replace_file_content` や `multi_replace_file_content` など）を利用してコードを修正します。

6. **修正後の検証**
   - 修正後に再度Lintやビルドを実行し、新たなエラーが発生していないか確認します。
   - 完了後、どのような修正を行ったかのサマリーを日本語で報告します。
